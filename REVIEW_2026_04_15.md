### Errores y aspectos a mejorar

#### **[IMPORTANTE] Emoticonos en presentación**

Aparecen emoticonos en los mensajes mostrados al usuario en `presentation/menu_bocadillos.py`:
- Línea 44: `"(✔)"` — marca de selección
- Línea 53: `"X "` — marca de error
- Línea 66, 69: `"✔"`, `"X "` — marcas de estado

Aunque funcionan, emoticonos como "✔" pueden causar problemas de compatibilidad con algunos terminales/codificaciones. Es preferible usar caracteres ASCII estándar.

**Cómo resolverlo:** Sustituye:
- `"✔"` → `"[OK]"` o `"[✓]"` 
- `"X "` → `"[ERROR]"` o `"[x]"`

Manteniendo el sentido del mensaje para que el usuario entienda si es éxito o error.

#### **[DISEÑO] `presentation/menu_bocadillos.py:92-104` — Acceso a servicios de usuario desde presentación**

La presentación accede directamente a `servicio_usuario.listar_usuarios()` para obtener la lista de autores disponibles. Aunque funciona, mezcla responsabilidades: la presentación debería ser pasiva (solo recopilar datos, mostrar resultados) y dejar que los servicios coordinen.

**Problema:** Si mañana cambia la forma de obtener autores (ej. desde una API remota), la presentación tendría que cambiar. La presentación **debe estar desacoplada** de cómo se obtienen los datos.

**Cómo resolverlo:** Considera pasar los autores disponibles como parámetro a `main_bocadillos()` desde el menú de ingredientes, O crear un método en `ServicioBocadillo` como `obtener_autores_disponibles()` que la presentación llama sin conocer los detalles de dónde vienen.

---

#### **[SUGERENCIA] Nombres de variables locales poco descriptivos**

En algunos métodos de presentación usas variables genéricas que podrían ser más específicas:
- `opcion` → `opcion_elegida` o `seleccion_menu`
- `seleccion` → `ingrediente_seleccionado`
- `autor_nombre` es bueno; mantenlo como referencia

Aunque el código funciona, nombres más descriptivos **mejoran legibilidad** cuando otros leen el código o lo revisas tras meses.

**Cómo resolverlo:** Revisa `presentation/menu_bocadillos.py` y `presentation/menu_ingredientes.py`, considera renombrar variables locales para que reflejen mejor qué contienen (ej. `indice_seleccionado` en lugar de `indice`).

---

#### **[SUGERENCIA] Docstring del módulo `domain/usuario.py`**

El fichero `domain/usuario.py` comienza con un comentario:
```python
# Más adelante tendrá más funcionalidades, pero ahora mismo sirve simplemente con el nombre
```

Esto debería ser un **docstring de módulo** (cadena entre comillas al inicio del fichero), no un comentario. Además, falta docstring en la clase `Usuario`.

**Cómo resolverlo:**
```python
"""
Módulo que define la entidad Usuario del dominio.

Un usuario representa un autor de bocadillos en el sistema.
"""

class Usuario:
    """
    Representa un usuario/autor en el sistema.
    
    :param nombre: Nombre del usuario (si está vacío se establece "Anónimo").
    """
```
